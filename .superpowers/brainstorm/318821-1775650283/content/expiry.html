How should the expiry work?

You want it to expire so people can't just share the HTML. "Not super safe, but enough that you need to be skillful to bypass."

🔑 Date-encoded token in URL
?token=boniches-2026-04-15-x7k2
• URL contains an expiry date, obfuscated
• JS decodes it and checks against today
• Expired = locked screen, no content
• You share a new URL per event/group
• Easy to bypass: change system clock or read JS

A — Token in URL

Unique URL per session/group with encoded expiry. Share boniches.html?t=abc123 for a specific date. No server needed.

🔓 Daily unlock code
Enter today's code: [ _ _ _ _ ]
• App shows a PIN entry screen on open
• Code changes daily (derived from date)
• You tell participants the code at the start
• Works offline, no URL parameter needed
• Easy to bypass: reverse the JS to find formula

B — Daily PIN code

App opens to a PIN screen. Code is date-derived (you can calculate it easily). Participants unlock it at the start of the activity. HTML can be shared freely — useless without today's code.

🔐 Token + hard expiry date
?t=x7k2 → valid Apr 10–15 only
• URL encodes a window (e.g. 3–7 days)
• After that window: locked, no content shown
• Content is also slightly obfuscated in HTML
• You generate a new token per event group
• Bypass requires JS knowledge + clock trick

C — Token + time window

URL contains a short-lived token valid for a specific date range (e.g. the day of the activity ± 1 day). After that it's locked. Content slightly obfuscated. Strongest of the three.
